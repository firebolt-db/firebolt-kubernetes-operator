# Known issues

Non-obvious problems, framework footguns, and environment traps that took meaningful debugging. See `AGENTS.md` for the policy on when to add to this file.

- **envtest's `kube-apiserver` ignores SIGTERM on macOS.** The embedded apiserver (>=1.35) does not act on SIGTERM on Darwin -- lease-renewal logs continue right up until envtest's SIGKILL fallback fires. Etcd handles SIGTERM cleanly, so this is apiserver-specific, not an envtest signalling bug. We can't fix it from this side; the workaround lives in two places: (1) `Makefile` sets `KUBEBUILDER_CONTROLPLANE_STOP_TIMEOUT=5s` only when `uname -s` reports `Darwin`, so SIGKILL fires fast instead of after the 20s upstream default; (2) `internal/controller/suite_test.go`'s `AfterSuite` swallows the resulting "timeout waiting for process kube-apiserver to stop" error, but only on `runtime.GOOS == "darwin"` and only for that exact error string. Linux/CI behaviour is unchanged -- a real teardown failure there still fails the suite. See controller-runtime#1571 / #2560 for upstream context.

- **`fsGroup` does not propagate to `subPath` mounts.** A pod with `runAsUser != 0` and a `volumeMount` that uses `subPath` will see the sub-directory as root-owned with only the group changed to `fsGroup`; the kubelet's recursive chgrp/chmod runs against the volume root, not the sub-path bind mount. This breaks any container whose entrypoint calls `chmod` against the mount (postgres' `chmod 00700 $PGDATA`, `initdb`'s mode enforcement, etc.) with `EPERM`. Resolution for the internal PostgreSQL pod (`internal/controller/instance_postgres.go`): mount the PVC root at `/var/lib/postgresql/data` without `subPath`, and set `PGDATA=/var/lib/postgresql/data/pgdata` so the postgres process creates and owns the data sub-directory itself. Upstream tracking: kubernetes/kubernetes#57923.

- **Engine binary refuses local-fs managed storage in dedicated-pensieve mode.** Starting with the metadata images shipped after 2026-05-13, the engine errors out at startup with "Invalid Firebolt configuration: local file system for managed storage is not supported when using dedicated Pensieve" unless `storage.bucket_name` is set (which maps internally to `firebolt.managed_storage.bucket_name_override`). The E2E suite (`test/e2e/floci_test.go`) and the helm-test scripts (`scripts/ci/lib/setup-floci.sh`) stand up [floci](https://github.com/floci-io/floci), a local S3 emulator, in the test namespace and pre-create a bucket; engines are then created with `spec.customEngineConfig.storage` set to `{type: minio, api_scheme: "s3://", bucket_name: <bucket>, minio.endpoint: http://floci.<ns>.svc:4566}`. `type: minio` flips the engine's `firebolt.aws.use_minio=true` flag, which hardcodes the S3 access/secret to `firebolt/firebolt` — floci is zero-auth so those baked-in credentials sign requests just fine, and no AWS env vars need to reach the engine pod. There is no `tmp_bucket_name` in the schema; managed and tmp tablets share `bucket_name`. Engine config reference: `src/Core/Application/Configuration/StorageConfig.h`.

- **`make lint` silently runs a degraded analyzer set when the golangci-lint cache (`~/Library/Caches/golangci-lint/` on macOS, `~/.cache/golangci-lint/` on Linux) is not writable.** Symptom: `make lint` exits `0 issues` locally but CI fails the same revision on a `goconst` / `gocritic` / `gosec` / similar finding. Cause: any agent harness that sandboxes filesystem writes (Cursor's default sandbox is one) blocks the cache writes; golangci-lint emits warnings like `level=warning msg="[linters_context/goanalysis] Failed to persist facts to cache: ... operation not permitted"` followed by `level=error msg="[pkgcache] cache close: ..."`, then proceeds with incremental fact-cache-dependent analyzers as no-ops. Those warnings are NOT benign — they mean the lint pass is incomplete. Resolution: when running `make lint` from inside a sandbox, request full filesystem access (in Cursor: `required_permissions: ["all"]` on the Shell tool call). The presence of any `Failed to persist facts to cache` or `[pkgcache] cache close` line in lint output means the result cannot be trusted and the command must be re-run with cache writes allowed before declaring the change green.

- **`controller-gen` DeepCopy fails on structs with `client.Reader` fields.** The `FireboltEngineCustomValidator` and similar validators hold a `client.Reader` interface so admission can do live API reads. controller-gen's `object` generator tries to synthesise `DeepCopyInto` for every type in the `api/v1alpha1` package, panics on the interface field, and fails the whole `make generate` run. Workaround: annotate the validator struct with `// +kubebuilder:object:generate=false` to opt it out of DeepCopy generation. See `api/v1alpha1/fireboltengine_webhook.go` for the canonical placement; apply the same marker to any future validator that needs a runtime client.

- **Full-fat CRDs overflow the Helm release Secret (1 MiB cap) and `helm install` fails.** Symptom: `helm install firebolt-crds ...` aborts with `Secret "sh.helm.release.v1.firebolt-crds.v1" is invalid: data: Too long: may not be more than 1048576 bytes`; the operator chart squeaks under at ~0.99 MB only because Helm stores `crds/` once (vs `templates/` twice) -- one new CRD field tips it over too. Cause: each CRD embeds a full `corev1.PodSpec` (the gateway / metadata / engine templates) whose OpenAPI schema is ~63% upstream Go-doc `description:` text; the three CRDs total ~2.5 MB raw, and Helm stores chart + rendered manifest (gzip+base64) in a Secret that Kubernetes hard-caps at 1 MiB. Resolution: `scripts/strip-crd-descriptions.py`, run by `make manifests`, writes description-slimmed copies into both charts' CRD dirs (`helm/firebolt-operator-crds/templates/` and `helm/firebolt-operator/crds/`), dropping only descriptions nested *inside* `template` sub-schemas (the upstream PodSpec docs, documented at kubernetes.io) while keeping Firebolt's own field docs for `kubectl explain`. Each release Secret drops to ~0.23 MB with zero effect on validation / CEL / pruning. **Ordering is load-bearing:** the strip MUST run after `patch-crd-template-metadata.py`, which keys off the description text "Standard object's metadata." to inject its `x-kubernetes-preserve-unknown-fields` markers (see the embedded-ObjectMeta pruning entry below) -- stripping first would erase that discriminator and silently reintroduce the labels/annotations pruning bug. `config/crd/bases` stays full-fat and canonical (kustomize, envtest, `kubectl explain`, json-schema artifacts). Guards: `internal/controller/crd_chart_slim_test.go` asserts each chart CRD equals its base minus descriptions; CI (`.github/workflows/lint.yaml`) re-runs `make manifests` and diffs the generated chart CRDs so a stale copy fails the build; and the `test-helm` workflow runs `make helm-test-crds` (`scripts/ci/verify-crd-chart-install.sh`), which actually `helm install`s the CRD chart on kind and fails if the release Secret is rejected -- the only guard that exercises the real install path.

- **A `+kubebuilder:default` on a `FireboltEngineSpec` field silently disables FireboltEngineClass inheritance for that field.** Symptom: a FireboltEngineClass sets `spec.rollout` (or `spec.drainCheckEnabled`), an engine that references the class leaves the field unset, yet the engine never picks up the class value. Cause: the apiserver applies the CRD default at admission and *stores* it on the engine, so by reconcile time the field is indistinguishable from an explicit user value. The `effective*` resolver follows engine-if-set → class-if-set → operator default, sees the materialized default as "engine set it," and short-circuits before reaching the class. Resolution: inheritable scalar/pointer fields carry NO CRD default. `rollout` and `drainCheckEnabled` have their defaults (graceful / true) applied last in the controller resolver (`effectiveRollout`, `effectiveDrainCheckEnabled` in `engine_reconcile.go`) instead. `storage` keeps `+kubebuilder:default={}` only because an empty `EngineStorageSpec` selects no backend, so it still falls through to the class. Tests: `internal/controller/engine_class_settings_test.go`. Adding a CRD default back to an inheritable engine field reintroduces this bug with no compile-time signal.

- **CRD pruning silently strips `template.metadata.labels` / `template.metadata.annotations` on embedded `corev1.PodTemplateSpec` fields.** Symptom: a user applies a FireboltEngineClass (or FireboltInstance gateway / metadata) with pod-template labels or annotations; the apiserver persists `metadata: {}`; any GitOps controller (Argo CD, Flux) re-applies the lost keys on every sync and drift never settles. Cause: controller-gen renders the embedded `metav1.ObjectMeta` inside `corev1.PodTemplateSpec` as `metadata: {type: object}` with no declared properties, and the structural-schema apiserver prunes everything not in the schema. The kubebuilder marker `+kubebuilder:pruning:PreserveUnknownFields` on the Go field lands at the `template` schema level — that does NOT propagate into a child node that has its own typed sub-schema, so the labels/annotations under `template.metadata` still get pruned. Resolution: `scripts/patch-crd-template-metadata.py` runs after `controller-gen` in the `manifests` target and injects `x-kubernetes-preserve-unknown-fields: true` directly on every embedded ObjectMeta stub (discriminator: the controller-gen description line "Standard object's metadata."). The regression-prevention test is `internal/controller/crd_pod_template_metadata_test.go` — a real-apiserver round-trip via envtest. The long-term fix is a custom `EmbeddedObjectMeta` type with explicit `labels` / `annotations` schema (à la prometheus-operator); this script is the cheap interim that keeps the API typed as `corev1.PodTemplateSpec`.

- **Bumping `k8s.io/*` to a release that raises the minimum Go (0.36 → Go 1.26) forces `golangci-lint` up in lockstep.** Symptom: a Dependabot PR that only bumps one `k8s.io` module (apimachinery, client-go, ...) fails the lint job at config load with `can't load config: the Go language version (go1.25) used to build golangci-lint is lower than the targeted Go version (1.26.0)` — even though the code compiles and tests pass. Cause: `k8s.io/*` 0.36 requires Go 1.26, so `go mod tidy` raises the `go` directive in `go.mod` to `1.26.0`; golangci-lint refuses to analyze a module whose `go` directive exceeds the Go version the linter itself was built with, and the pinned `v2.5.0` binary is built with go1.25. Resolution: bump the linter to the first release built with the new Go (v2.9.0 is the first go1.26 build) in **both** pins — `GOLANGCI_LINT_VERSION` in the `Makefile` and the `version:` input in `.github/workflows/lint.yaml`. The k8s ecosystem must also move as one unit: `controller-runtime` 0.24 pairs with `k8s.io` 0.36, and the Makefile derives `ENVTEST_VERSION` (`release-0.24`) and `ENVTEST_K8S_VERSION` (`1.36`) from `go.mod`, so a partial bump leaves envtest and controller-runtime skewed. Note the CI lint job installs golangci-lint as a prebuilt **binary** (the `golangci-lint-action`), whereas `make lint` uses `go install`; a local `go install` may hit unrelated module-checksum errors under `GOPROXY=direct` that CI never sees — verify locally with the prebuilt binary instead.

- **controller-runtime v0.24 made the webhook builder generic and deprecated `client.Apply` / `GetEventRecorderFor` / `scheme.Builder`.** Symptom 1 (compile break): `ctrl.NewWebhookManagedBy(mgr)` fails with `not enough arguments ... want (manager.Manager, T)`. The builder is now `WebhookBuilder[T runtime.Object]`; pass the object to the constructor (`ctrl.NewWebhookManagedBy(mgr, &FireboltEngine{})`) and drop the old `.For(&FireboltEngine{})` call. `WithValidator`/`WithDefaulter` now take `admission.Validator[T]`/`admission.Defaulter[T]` (the old `CustomValidator`/`CustomDefaulter` are deprecated aliases for `…[runtime.Object]`). **Footgun:** instantiating with `T = runtime.Object` to keep the old `runtime.Object`-typed validator signatures *compiles* but panics at admission time — the generic handler builds its decode target via `reflect.TypeOf(zero T)`, and `reflect.TypeOf` of a nil interface is nil. Use the concrete pointer type (`*FireboltEngine`) so `T` is a concrete pointer; this also lets the validator/defaulter methods take the typed object directly and drop the `obj.(*T)` assertion (which `forcetypeassert` would otherwise flag). See the three `api/v1alpha1/*_webhook.go` files for the pattern. Symptom 2 (lint): `client.Apply` (the SSA-via-`Patch` sentinel, ~14 sites), `manager.GetEventRecorderFor`, and the api-package `scheme.Builder` are deprecated under SA1019. They remain fully functional in 0.24; their replacements (typed `Client.Apply` with ApplyConfigurations, the new events API, direct `runtime` scheme registration) are behaviour-sensitive migrations deferred out of the dependency bump and suppressed via scoped `text:` exclusions in `.golangci.yml`.
